# Lessons

Patterns from corrections. One entry per lesson.

- 2026-06-09 — "didn't worked" after feature shipped: all verification ran through pytest (`pythonpath=["src"]`) and PYTHONPATH-injected smokes, so a stale non-editable site-packages install + missing Scripts dir on PATH silently shadowed every new command on the user's real shell.
  **Rule: before declaring CLI work done, run the exact documented command in a fresh shell the way the user would (installed entry point, no PYTHONPATH crutch). If the env is stale, fix the env (editable install) as part of the task — not as a "follow-up" note.**
- 2026-06-09 — `python -m package` execution is the PATH-proof fallback; ship a package `__main__.py` for any CLI so docs can offer it when the console script isn't resolvable.
- 2026-06-09 — "NOT WORKING" after a UI fix traced to two server bugs, not the UI: (1) the server cached the page at startup, so a long-running instance kept serving stale HTML across updates — read assets per request in dev-facing servers; (2) stdlib HTTPServer's SO_REUSEADDR lets two processes silently bind the same port on Windows — bind with SO_EXCLUSIVEADDRUSE there and surface 'port in use' clearly.
  **Rule: when a user reruns a long-lived local server across code updates, ask first: which process is actually serving, and which version is in its memory? Check the live process list and the served bytes, not the source files.**
- 2026-06-10 — "Output format duckdb has no registered output handler" on first real run: the new handler was registered in `templates/data_curator/__main__.py`, but runs execute the user's scaffolded workspace copy (repo-root `__main__.py`), which `start` deliberately never overwrites — so template changes silently don't reach existing workspaces.
  **Rule: when changing the entry-script template, also update every live workspace copy the user actually runs (repo root here), and smoke-test through the panel's real run path — not just the library tests. Registering a new handler/provider touches: handler module, `__init__`, template, workspace `__main__.py`, `OUTPUT_FORMATS`, docs.**
- 2026-06-10 — DuckDB crash on first real S&P 500 run: pyarrow infers minimal decimal precision per batch (`Decimal('0')` column → `decimal128(1,0)`), so the first ticker's degenerate type got locked into the shared table schema and the next ticker's real values overflowed (`DECIMAL(1,0)` vs 104773000). Synthetic float-only test tables never exposed it.
  **Rule: any schema shared across batches must reconcile types per write, not trust the first batch; widen via the engine's own UNION super-type inference. And test accumulation features with realistic type variance: decimals, all-null columns, per-batch precision drift — not just clean doubles.**
- 2026-06-17 — `/deep-research` to enumerate provider series IDs mostly failed: 108 agents / 3.3M tokens produced ONE usable new series. Every DBnomics claim scored 0-0 ("killed"), but the failures log was wall-to-wall `Server is temporarily limiting requests · Rate limited` — the verifier's WebFetch hit Anthropic's web-infra rate limit, and the synthesizer wrongly treated rate-limit-failure as refutation. A plain `curl` to the same DBnomics API returned every series instantly.
  **Rule: generic web-search deep-research is the wrong instrument for enumerating/verifying structured API identifiers. When the source is a queryable API (DBnomics keyless, FRED `fredgraph.csv` keyless), verify by hitting the API DIRECTLY (curl/httpx in a deterministic script), not via web-search agents. Distinguish "refuted" from "could-not-check" — a 0-0/abstain vote on a verifiable fact is an access failure to retry, never evidence the claim is false. For bulk catalog work, a verifier script (existence-check each id, keep only live ones, emit an audit trail) beats an agent swarm on cost, reliability, and reproducibility.**
- 2026-06-17 — `httpx` intermittently `ReadTimeout`ed on FRED's `fredgraph.csv` first request (and a custom `User-Agent` made it worse), while `curl` was 100% reliable for the same URL.
  **Rule: when an httpx call to an external host flakes/stalls but curl works, shell out to curl for that host rather than fighting the client; and check `content-type` substrings loosely (FRED serves `application/csv`, not `text/csv`).**
- 2026-06-24 — Macro-only export crashed in DuckDB (`Binder Error: Referenced column "m_date" not found`) only on a DB that already held tickers. The standalone-export feature shipped a `(date, value)` table tested against CSV/Parquet/in-memory-store only; the DuckDB upsert keys on the `(main_identifier, m_date)` PK and a fresh DB silently fell back to PK-less wholesale-replace, so no test or smoke ever exercised the keyed path. `m_date` is a cross-handler contract (every other output emits it; DuckDB keys on it, in-memory `export_dataframe` indexes on it) — a new producer using a bare `date` passes the lenient handlers and breaks only the stateful one, against pre-existing state.
  **Rule: a new output-table SHAPE must be exercised through EVERY registered handler, and the stateful handler (DuckDB) against a PRE-EXISTING table, not just a fresh one — the empty-DB path hides PK/upsert mismatches. Reuse the system's canonical column names (`m_date` for the date axis) rather than inventing a local one; if a deviation is deliberate, it owes a test on every handler that keys/indexes on that column. Mirrors the 2026-06-10 DuckDB lesson: shared/stateful schema bugs only surface under realistic prior state.**
