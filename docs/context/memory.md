# Memory

Architectural decisions and durable context. One line per item.

- decision (2026-06-09): JSON config + local HTML editor (`config-editor` CLI) replaces Excel as the default configuration path; `ExcelConfigurator` kept as a deprecated, working fallback via the `ConfiguratorInterface` seam.
- decision (2026-06-09): shared post-parse logic for configurators lives in `config_handlers/_resolver.py`; `JsonConfigurator` uses it, `ExcelConfigurator` left untouched (follow-up: refactor Excel to delegate behind characterization tests).
- decision (2026-06-09): output column catalog is a bundled `config_handlers/column_catalog.json` (201 columns, 8 prefix groups), generated from the xlsx; single source of truth for the editor picker.
- decision (2026-06-09): `start` is the single user-facing command — non-destructive workspace scaffold + panel + in-panel "Save & run" (`POST/GET /api/run`, background thread, output captured at completion); `init json`/`config-editor`/`run` remain as granular commands.
- decision (2026-06-10): repo is now a private standalone at github.com/alanvaa06/Data-Curator (fork deleted, no KaxaNuk remote); MIT license retained.
- decision (2026-06-10): Excel configuration fully removed (ExcelConfigurator, autorun, init/update excel, xlsx template, panel import-excel, openpyxl dep) — JSON + panel is the only config path; templates/__main__.py is the JSON entry script.
- decision (2026-06-10): configurators raise (ConfigurationError/ConfigurationHandlerError) instead of sys.exit; entry-script template catches DataCuratorError and exits 1.
- decision (2026-06-10): mypy is a blocking CI step; legacy modules (~1200 errors, mostly fmp/lseg god-modules) baselined via ignore_errors overrides in pyproject — ratchet down over time.
- decision (2026-06-10): 'duckdb' output format writes all identifiers to one curated_data table keyed (main_identifier, m_date) via INSERT OR REPLACE — incremental refreshes update in place; csv/parquet remain the per-ticker interop formats; fetch side still downloads the full configured range (incremental fetch is a follow-up).
- decision (2026-06-17): macro data layer selected = direct national providers (Banxico SIE + INEGI, the only adversarially-verified Mexico-gate coverage), NOT an aggregator (DBnomics/EconDB unverified, refuted on INEGI) and US layer = FRED (optional, bring-your-own-key). Re-evaluated 2026-06-17: FRED's June-2024 ToU (no storing/caching/incorporating-in-database, no ML training) binds the END-USER key holder, not this MIT tool which ships code-not-data; FRED endorses research/academic use + apps that interoperate with the API, and Fair-Use targets bulk REDISTRIBUTION not personal storage — so FRED is fine for a non-commercial OSS tool (same pattern as fredapi/pandas-datareader). FRED uniquely offers ALFRED point-in-time vintages (reproducibility). Caveat: verbatim ToU unfetched (FRED bot-blocks); flips back to disqualified if KaxaNuk ships FRED data commercially or trains models on it. Rest-of-world layer = DBnomics (keyless, OECD/WB/IMF/Eurostat/ECB/BIS; no FRED, no Banxico) or World Bank direct (CC-BY 4.0). MX still direct (Banxico+INEGI). Integration = new EconomicIndicatorDataBlock with grouping_identifier_field=None (macro broadcasts to all tickers, the block system's intended first non-equity use) + sibling MacroDataProviderInterface + e_* columns forward-filled to market dates; existing output unchanged. Proposal at docs/superpowers/specs/2026-06-17-macro-data-layer-proposal.md; build gated on Phase-0 verifications.
- decision (2026-06-17): macro catalog is DATA-DRIVEN (curated JSON mapping each e_* column → provider + provider_series_id, internal routing; user picks named columns like equity columns). Locked v1 design choices: (a) curated catalog w/ routing, (b) latest-values-only — NO vintages in v1 (date→value entity; FRED/ALFRED vintages deferred), (c) thin direct HTTP for ALL adapters (Banxico/INEGI/FRED/DBnomics) reusing pooled httpx — ZERO new runtime deps. v1 starter catalog: US/FRED (cpi, core_cpi, fed_funds, 2y, 10y, unemployment, gdp, m2), MX/Banxico (target_rate, tiie28, cetes28, usdmxn_fix, reserves), MX/INEGI (inpc, gdp, unemployment — exact IDs are open Phase-0 gate), global/DBnomics (ez_hicp, ecb_rate). Catalog MUST be built to expand to a comprehensive set later (user requirement).
- reference (2026-06-17): macro source COMMERCIAL-redistribution status (for KaxaNuk's future commercial product gate) — COMMERCIAL-OK: World Bank Open Data (CC-BY 4.0, attribution required; some 3rd-party series carved out) [verified]. RESEARCH/PERSONAL-ONLY, NOT cleanly commercial: FRED (June-2024 ToU restricts storing/caching/redistribution + ML/LLM training; many series carry 3rd-party copyright — fine for this OSS BYO-key research tool, BLOCKED for a commercial product or model-training without per-series clearance) [verified]. UNVERIFIED, confirm before any commercial use: Banxico SIE + INEGI (likely public-with-attribution under Mexico's Ley del SNIEG, not yet read), OECD + IMF (generally reuse-with-attribution). DBnomics itself = open aggregator but a PASSTHROUGH — the underlying provider's license governs each series.
- reference (2026-06-17): INEGI macro series IDs (Phase-0 gate, partially closed via research) — INPC headline CPI index level = indicator `216064` (HIGH conf, BIE bank); national unemployment/ENOE = `444612` (MED-HIGH, BIE); real GDP/PIB = `381016` (LOW conf — NOT confirmed as quarterly-real-national-total, verify via authenticated round-trip reading Series[0].FREQ); core/subyacente INPC = UNRESOLVED (don't guess). All BIE bank (not BISE), so adapter URL uses `/es/00/false/BIE/2.0/{token}`. JSON: Series[0].OBSERVATIONS[].{TIME_PERIOD,OBS_VALUE}; OBS_VALUE is a string, missing = None or ""; TIME_PERIOD format depends on FREQ (8=monthly "YYYY/MM", 6=quarterly "YYYY/0Q"); observations come NEWEST-first → sort ascending; token required (404 without). For commercial use note INEGI licensing still unverified.
